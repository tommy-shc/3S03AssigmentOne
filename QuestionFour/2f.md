TDD is a development strategy we would use selectively depending on the context. It is well suited for well-defined, isolated functionality like the calculator's divide() method, where the expected behaviour and edge cases can be clearly specified upfront. In a larger system with tight coupling or rapidly changing requirements, writing tests first can become difficult since the expected behaviour may not yet be fully understood.

The element of TDD we found most valuable is that it forces you to think critically about edge cases and expected behaviour before writing any implementation. This allows us to have a more planned out development process and the resulting code more reliable.

The hardest element to manage in TDD is knowing when your test suite is complete enough to move on. As seen in this exercise, it is easy to miss edge cases in the first iteration, and there is no clear sign telling you when you have tested enough. This requires experience and domain knowledge that can be difficult to apply consistently, especially on a large or unfamiliar codebase.
